# Contribuir a Rubik OS

## Principios de trabajo

- Trabajar a partir de requisitos y reglas de negocio validados.
- Proponer y justificar cambios de arquitectura antes de implementarlos.
- Mantener cambios pequeños, claros, probados y documentados.
- Usar migraciones versionadas para todo cambio de base de datos.
- No incluir secretos, credenciales ni datos personales en Git.
- Registrar decisiones arquitectónicas relevantes en `docs/adr/`.

## Flujo de cambios

1. Crear una rama de trabajo desde la rama principal actualizada.
2. Implementar el cambio con sus pruebas y documentación.
3. Verificar localmente formato, análisis estático y pruebas.
4. Abrir un pull request utilizando la plantilla del repositorio.
5. Obtener las revisiones requeridas antes de integrar.

Las convenciones específicas de ramas, commits y herramientas se documentarán cuando sean aprobadas.
